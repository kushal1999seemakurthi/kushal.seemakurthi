# 🧊 Immersive WebGL Architecture: The "Igloo" Blueprint

> ⚠️ **Stack Reference Updated.** This document references **Svelte** as the frontend framework and **Three.js** directly. ImmersivePort uses **React 19 + React Three Fiber (R3F)** instead. Core concepts about scroll-scrubbing via a single `0..1` progress float, shader-driven UI, GPGPU particles, instanced rendering, and strict memory management remain valid and apply directly. Adapt all component lifecycle patterns to React `useEffect` / `useFrame` hooks rather than Svelte component teardown.

> **Agent Directive:** This document serves as the foundational architecture and concept guide for building a high-performance, scroll-driven 3D web experience. Your objective is to use these principles to guide component generation, shader authoring, and state management, treating the browser as a real-time rendering engine rather than a static document viewer.

## 🛠️ 1. The Tech Stack

This architecture heavily bypasses the traditional DOM in favor of GPU-accelerated rendering. 

* **Core 3D Engine:** `Three.js`
* **Frontend Framework:** `Svelte` (Chosen specifically for its lightweight, compiler-based footprint and lack of a virtual DOM, making it ideal for tight, lag-free WebGL synchronization).
* **Animation & State:** `GSAP` (GreenSock) + `ScrollTrigger` (For timeline scrubbing and interpolation).
* **Build Tool:** `Vite` (Essential for lightning-fast Hot Module Replacement during shader development).
* **Performance Optimization:** `three-mesh-bvh` (Optimizes raycasting against complex 3D geometries, crucial for hover interactions).
* **Asset Pipeline (Off-browser):** `Houdini` (for procedural generation, like simulating ice crystal growth) and `Blender`.

## 🧠 2. Core Concepts & Technical Practices

### 2.1. Scroll-Scrubbing (The "Progress" Variable)
**Rule: Do not use Intersection Observers for continuous motion.**
* **Concept:** Map the entire page's vertical scroll height to a normalized progress float (0.00 to 1.00). 
* **Implementation:** Feed this single variable into a master GSAP timeline. As the user scrolls, the timeline is mathematically "scrubbed" forward and backward. This single data stream drives the camera spline path, object rotations, and shader uniform transitions simultaneously.

### 2.2. Shader-Driven UI
**Rule: Avoid DOM reflows for interactive visuals.**
* **Concept:** Traditional HTML/CSS is too slow for complex, frame-by-frame glitch or dissolve effects.
* **Implementation:** Render UI elements and typography directly in WebGL. Use Signed Distance Fields (SDF) for crisp text rendering at any scale. Handle text glitches or letter scrambles by swapping SDF texture offsets via custom GLSL shaders, executing the logic entirely on the GPU.

### 2.3. GPGPU Particles & Volume Data
**Rule: Offload physics and particle calculations from the CPU.**
* **Concept:** For complex particle systems (like interactive, swirling footers), the CPU cannot calculate thousands of trajectories at 60fps.
* **Implementation:** Utilize General-Purpose computing on Graphics Processing Units (GPGPU). Store particle positions, velocities, and lifetimes in Framebuffer Objects (FBOs) as texture data. Update them frame-by-frame via fragment shaders.
* *Data Pipeline:* To achieve complex particle formations, export VDB volume data from external 3D software, compress it into binary or lightweight textures, and parse it in the browser to dictate particle attractor points.

### 2.4. Procedural Generation Algorithms
**Rule: Scale assets programmatically, not manually.**
* **Concept:** Instead of loading massive pre-modeled static assets, generate them mathematically at runtime or through lightweight baked algorithms.
* **Implementation:** Write functions that simulate physical processes (like crystal growth) inside a bounded 3D container shape. This ensures every instantiated object is unique without inflating the initial payload size.

## 🤖 3. Agent Execution Directives

When generating or refactoring code for this repository, you must strictly adhere to the following protocols:

1.  **Canvas Supremacy:** The application layout should primarily consist of a single full-screen `<canvas>` element. Standard HTML/CSS should be strictly reserved for semantic accessibility overlays, invisible scroll-containers, or static loading states.
2.  **Logic/Render Separation:** Keep Svelte components focused strictly on lifecycle management and state reactivity. Isolate all heavy 3D math, scene initialization, and requestAnimationFrame loops into dedicated, modular TypeScript classes.
3.  **Strict Memory Management:** Three.js does not garbage collect materials and geometries automatically. You must explicitly author `.dispose()` methods for all geometries, materials, and textures, and call them automatically during component teardown to prevent memory leaks.
4.  **Shader Modularity:** Write GLSL shaders in separate `.glsl` or `.vert`/`.frag` files using `vite-plugin-glsl`. Do not output inline string shaders. Ensure modularity and reuse.
5.  **Instanced Rendering Only:** Whenever generating code for multiple identical geometries (debris, particles, repeating UI blocks), you must implement `InstancedMesh`. Never loop and create individual mesh instances, as this will aggressively inflate draw calls.
